## 1. 用户需求记录（原始需求 + 收口约束）
- 需求描述：
  - 导入选择文件夹，批量扫描文件夹内模型数据，并导入模型库
  - 项目配置监控目录，定期自动监控新增文件进行模型导入
  - 记录日志，完成后推送汇总信息
- 用户最终收口要求：
  - 归档模型和本地导入模型完全独立，不能混在一起
  - 本地导入模型可增加模型 / 配置标记字段，归档模型不需要
  - 批量导入逻辑单独拆文件，不继续堆在 `server.py`
  - 监控目录必须有页面开关，关闭时不启动线程
  - 默认监控目录设为 `./watch`
  - 成功 / 重复文件移动到指定目录并跳过后续扫描
  - 失败文件移动到另一个目录并跳过后续扫描
  - 手动导入页面要增加批量目录导入入口
  - 无更新扫描不生成报告文件

## 2. AI 分析（最终实现方案）
- 实现难度：中
- 最终实现思路：
  - 新增独立批量导入服务层，复用现有本地导入落盘逻辑
  - 只在 `LocalModel_*` 范围内做模型聚合与配置去重
  - 监控目录使用轮询扫描，不使用文件系统事件监听
  - 解析可并发，落盘与目录写入保持串行
- 关键约束落地：
  - `MW_*` 在线归档与 `LocalModel_*` 本地导入完全隔离
  - 本地批量导入元数据仅写入本地模型 `meta.json`
  - watcher 仅在开关开启时运行

### 3.2 本次实际实现能力
- 手动目录批量导入：
  - 手动导入弹窗可以直接选择目录上传并导入
- 监控目录批量导入：
  - 配置页可开启监控目录轮询
  - 默认目录为 `./watch`
  - 支持递归扫描子目录中的 `3MF`
- 本地模型聚合规则：
  - 基于 `modelKey + configFingerprint` 做模型 / 配置判定
  - 同模型不同配置聚合到同一个 `LocalModel_*`
  - 同模型同配置重复导入时跳过
- 文件处理规则：
  - 成功 / 重复文件移动到 `_imported/`
  - 失败文件移动到 `_failed/`
  - 后续扫描忽略 `_imported/` / `_failed/`
  - 移动完成后自动清理原路径空文件夹
- 结果展示与推送：
  - 配置页显示最新扫描记录
  - 配置页显示最近 5 次成功记录
  - 显示处理耗时、处理数、成功 / 跳过 / 失败统计
  - 推送标题区分“监控目录导入完成”和“手动目录导入完成”

## 4. 与初版预期差异
- 原始分析中存在“预扫描 + 再确认导入”的两阶段思路
- 实际实现改为“直接执行导入 + 结构化报告”
- 原因：
  - 当前需求更强调稳定落地和不影响现有逻辑
  - 直接执行模式更适合监控目录和大量文件处理

## 6. 用户使用建议
- 少量文件：
  - 使用手动导入弹窗中的“选择目录批量导入”
- 大量文件：
  - 优先把文件先放到 `watch` 目录
  - 再到配置页点击“立即扫描并导入”
  - 这样能减少浏览器上传等待时间，也更适合 Docker 映射目录
