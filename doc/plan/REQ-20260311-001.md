## 1. 用户需求记录（原始需求 + 初步思路）
- 需求描述：
  - 增加一个本地 Python 脚本，用于一键整理指定文件夹中的全部 `3mf` 文件。
  - 整理规则为“一个模型一个文件夹”，同模型下的不同配置放在同一个模型文件夹内。
  - 需要根据配置名重命名文件名，便于后续人工查看和管理。
  - 如果遇到重复文件，需要单独集中放到根目录下另一个重复文件夹中。
  - 整理完成后输出文本总结，至少包含模型数、配置数、重复数。
  - 后续希望这个能力可以嵌入当前页面，固定读取一个本地目录，点击页面按钮后执行整理。
- 用户初步实现思路：
  - 先做独立的本地脚本，支持直接双击或命令行一键运行。
  - 脚本读取文件夹中的全部 `3mf`，解析模型名和配置名，再按规则重新归类。
  - 页面侧后续增加一个固定目录和“开始整理”按钮，直接调用同一套整理逻辑。
- 目标效果：
  - 原始散乱的 `3mf` 文件可被自动整理成稳定目录结构。
  - 相同模型的不同配置自动归档到一起。
  - 重复文件不会混入主目录，而是被集中隔离出来。
  - 每次运行都有可追踪的文本报告，便于核对。
- 约束/注意事项：
  - 脚本必须使用 Python。
  - 当前阶段先做需求分析文档，不直接开始编码。
  - 后续页面接入需要尽量复用脚本核心逻辑，避免写两套规则。

## 2. AI 分析（难度 / 思路 / 改动范围）
- 实现难度：中
- 当前代码基础：
  - `app/three_mf_parser.py` 已具备读取 `3mf` 包、提取 `Title`、`ProfileTitle`、`DesignModelId`、`DesignProfileId` 等元数据的能力，可直接复用解析逻辑。
  - `app/batch_import_service.py` 已实现“按模型聚合、按配置判重、输出报告”的批量导入框架，其中 `derive_model_key()` 与 `derive_config_fingerprint()` 的规则可以直接借鉴。
  - `app/server.py` 已有本地批量导入接口 `/api/local-batch-import/*`，说明当前页面已具备“按钮触发本地文件处理”的接入点，后续可以新增一个平行的“整理目录”能力。
  - `scripts/` 目录目前已有多个独立 Python 脚本，适合新增本次整理脚本。
- 需求本质判断：
  - 这次不是“导入到模型库”，而是“先在文件系统层面做预整理”。
  - 更适合设计为两层：
  - 第 1 层：纯本地整理器，只负责扫描、归类、重命名、判重、输出总结。
  - 第 2 层：页面入口或后端接口，只负责调用第 1 层并展示结果。
  - 这样后续无论独立运行还是页面触发，都共用一套核心规则。
- 推荐目录结构：
  - 假设输入根目录为 `input_root/`
  - 整理后建议输出到固定子目录，例如：
  - `input_root/整理完成/<模型目录>/<配置文件>.3mf`
  - `input_root/重复文件/<原始文件或冲突文件>.3mf`
  - `input_root/整理报告/organize_report_YYYYMMDD_HHMMSS.txt`
  - 不建议直接在原始根目录平铺改名，最好统一进入 `整理完成`，这样更安全，也方便回滚。
- 推荐命名规则：
  - 模型文件夹名：
  - 有 `DesignModelId` 时使用 `MW_<designer>_<modelTitle>`，不再显示 `DesignModelId`
  - 缺失 `DesignModelId` 时使用 `Others_<designer>_<modelTitle>`，若没有作者则退化为 `Others_<modelTitle>`
  - 文件夹显示名建议保留可读中文标题，但内部判定 key 使用稳定字段。
  - 配置文件名：
  - 优先使用 `profileTitle`
  - 若缺失，则退化为原始文件名去扩展名
  - 输出为 `<配置名>.3mf`
  - 若同一模型目录内出现同名但不同内容的配置冲突，建议自动追加序号，如 `<配置名>_2.3mf`
- 推荐判重规则：
  - 模型归类 key：
  - 优先 `DesignModelId`
  - 退化为 `规范化 modelTitle + designer`
  - 配置判重 key：
  - 优先 `DesignProfileId`
  - 若缺失，则使用文件 `sha256`
  - 这个规则与现有 `app/batch_import_service.py` 一致，稳定性较高。
  - 对“重复”的定义建议明确分成两类：
  - 完全重复：同模型 key 且同配置 key，直接归入 `_duplicates`
  - 命名冲突但内容不同：不是重复，属于同模型下的新配置，需要自动改名落入模型目录
- 推荐输出结果：
  - 控制台输出简版结果。
  - 同时写入文本报告，便于用户保存和复查。
  - 报告建议包含：
  - 本次扫描总文件数
  - 成功整理的模型数
  - 成功整理的配置数
  - 完全重复数
  - 解析失败数
  - 每个模型下收纳了哪些配置
  - 重复文件清单
  - 失败文件清单
- 对“一个模型一个文件夹”的理解：
  - 这里更接近“按设计模型聚类”，不是按原始文件名分组。
  - 如果多个 `3mf` 的 `Title` 相同但 `DesignModelId` 不同，原则上应视为不同模型。
  - 如果 `DesignModelId` 缺失，只能退化为标题规则，这时会有误归类风险，需要在文档和报告中标明“弱匹配”来源。
- 与现有批量导入的关系：
  - 现有 `batch_import_service` 是把 `3mf` 导入到 `app/data/LocalModel_*` 结构，并生成 `meta.json`、图片、附件等模型库数据。
  - 本次脚本只做“源文件预整理”，不写入模型库，不生成 `meta.json`，不影响现有归档数据。
  - 后续可以形成串联流程：
  - 先执行本地整理脚本，把散乱文件归档到 `_organized`
  - 再由页面上的现有“本地批量导入”从该目录继续导入到模型库
  - 这比直接让导入模块兼顾“文件系统整理”更清晰，也更容易排查问题。
- 推荐实现拆分：
  - 阶段 1：独立脚本
  - 新增例如 `scripts/organize_local_3mf.py`
  - 支持传入目标目录，默认可指向一个固定目录
  - 支持 `--dry-run` 预览模式
  - 支持生成 `.txt` 报告
  - 阶段 2：页面接入
  - 配置中增加固定整理目录，例如 `local_3mf_organizer.watch_dir`
  - 后端新增接口，例如 `POST /api/local-3mf-organizer/run`
  - 页面增加按钮“整理固定文件夹”
  - 页面展示最近一次整理结果和报告路径
- 脚本运行方式建议：
  - 基础版：
  - `python scripts/organize_local_3mf.py --root "D:\\xxx\\folder"`
  - 一键版：
  - 可选增加同目录 `.bat` 或 `.ps1` 包装脚本，但核心逻辑仍保持在 Python 中
  - 如果用户强调“双击即用”，可以在后续实现阶段补一个 `run_organize_local_3mf.bat`
- 关键设计取舍：
  - 是否直接原地改动原文件：
  - 不建议。应优先“复制或移动到新目录”，降低误操作风险。
  - 是否默认移动还是复制：
  - 若目标是“整理本地存量文件”，更偏向移动。
  - 但为了安全，第一版建议支持 `--mode copy|move`，默认 `copy` 或至少先做 `dry-run`。
  - 是否扫描子目录：
  - 用户描述是“读取文件夹内所有 3mf 文件”，建议默认递归扫描。
  - 但要自动排除 `_organized`、`_duplicates`、`_reports` 等输出目录，避免重复处理。
- 主要风险点：
  - 部分 `3mf` 可能没有完整元数据，导致模型名或配置名提取不稳定。
  - 仅用标题匹配的场景可能会把不同模型错误合并，需要在报告中标出匹配来源。
  - 中文、空格、特殊字符、超长路径在 Windows 下都需要做文件名清洗和长度控制。
  - 若直接“移动”原文件，一旦中途中断可能造成目录处于半完成状态，因此要么采用事务式流程，要么先复制后统一清理。
  - 后续页面触发本地整理时，如果服务端运行在 Docker 中，容器内路径和宿主机路径需要单独定义，不能直接假设与当前桌面环境一致。
- 验证方案：
  - 准备一组测试样本：
  - 同模型不同配置
  - 同模型同配置重复文件
  - 同模型同配置但不同原始文件名
  - 无 `DesignModelId` / 无 `DesignProfileId` 的文件
  - 含中文和特殊字符的文件名
  - 空文件或损坏 `3mf`
  - 验证输出目录是否满足“一个模型一个文件夹”。
  - 验证同模型下配置文件是否按配置名重命名。
  - 验证完全重复文件是否全部进入 `_duplicates`。
  - 验证总结文本中的模型数、配置数、重复数、失败数是否准确。
  - 验证重复运行同一目录时不会把输出目录再次扫进去。
