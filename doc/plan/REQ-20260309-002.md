## 1. 用户需求记录（原始需求 + 初步思路）
- 需求描述：
  - 首页 UI 增加“多选模式”，支持在选中多个模型后执行批量操作。
  - 引入“收藏夹”功能，可创建收藏夹，填写收藏夹名称和简介，并支持将多选模型加入收藏夹。
  - 引入“批量删除”功能，多选后可批量删除，删除前需要二次确认。
  - 引入“简洁模式”切换：首页封面图略微放大，卡片只显示模型名称。
- 用户初步实现思路：
  - 先在首页打开多选，再基于选中结果执行“加入收藏夹”或“批量删除”。
  - 收藏夹信息持久化到 `config` 目录下的 JSON 文件。
  - 简洁模式作为首页视图切换，不影响模型详情页。
- 目标效果：
  - 首页从“单卡片逐个操作”升级为“单选操作 + 多选批处理”并存。
  - 收藏夹可以作为新的归类手段，便于整理大量模型。
  - 批量删除降低重复点击成本，但保留安全确认。
  - 简洁模式提升浏览密度，适合快速扫模型。
- 约束/注意事项：
  - 不能破坏当前单模型收藏、已打印、删除、筛选和排序行为。
  - 多选模式要兼容现有无限滚动，不应因为重新渲染导致选中状态丢失。
  - 批量删除属于物理删除，必须做明确确认和异常反馈。
  - 收藏夹命名与简介需要有基础校验，避免空名称和重复命名冲突。

## 2. AI 分析（难度 / 思路 / 改动范围）
- 实现难度：中
- 当前代码基础：
  - 首页模板在 `app/templates/gallery.html`，已经有顶部工具区、筛选侧栏、卡片网格和卡片底部操作区。
  - 首页交互在 `app/static/js/gallery.js`，当前已支持搜索、筛选、无限滚动、单模型收藏、已打印标记、单模型删除。
  - 后端首页数据由 `app/server.py` 中的 `scan_gallery()` 生成，接口入口为 `/api/gallery`。
  - 首页标记数据当前持久化在 `app/config/gallery_flags.json`，结构为 `favorites` 与 `printed` 两组数组。
- 总体实现思路：
  - 以“选择状态”作为首页新增的前端状态层，加入 `selectedModelKeys`、`selectionMode`、`compactMode` 三类状态。
  - 保持现有单卡片按钮可用，同时增加顶部批量操作条；未开启多选时维持当前首页使用习惯。
  - 收藏夹与首页标记同属首页维度状态，建议直接扩展现有 `gallery_flags.json`，而不是再新建零散文件，降低迁移与读写复杂度。
  - 批量删除建议新增独立后端接口，避免前端循环调用单删接口导致部分成功、部分失败时难以汇总反馈。
  - 简洁模式优先做成前端视图模式，建议默认用 `localStorage` 记忆，避免影响其他访问者；如果后续确认需要多人共享，再上升到服务端配置。
- 推荐数据结构：
  - 继续使用 `app/config/gallery_flags.json`，在现有结构基础上扩展为：
  - `favorites: string[]`
  - `printed: string[]`
  - `folders: [{ id, name, description, modelDirs: string[], createdAt, updatedAt }]`
  - 可选预留 `ui: { compactMode?: boolean }`
  - 兼容策略：`load_gallery_flags()` 读取旧文件时自动补默认字段；`save_gallery_flags()` 输出新结构并去重。
- 首页交互方案：
  - 顶部工具栏新增“多选”开关，开启后卡片进入可选中状态，卡片左上角显示复选框或选中蒙层。
  - 顶部工具栏在有选中项时切换为批量操作条，显示“已选 N 项”“加入收藏夹”“批量删除”“取消选择”。
  - 卡片点击行为在多选模式下需要区分：
    - 点击复选区为选中/取消选中。
    - 点击标题或封面默认不再直接打开详情，避免误触；建议改为提供独立“查看详情”入口，或仅在未开启多选时保持跳转。
  - 无限滚动下重渲染时，选中状态由 `selectedModelKeys` 统一回填。
- 收藏夹功能方案：
  - 首页新增收藏夹入口，支持新建收藏夹弹窗，字段为“名称”“简介”。
  - 收藏夹应允许：
    - 新建后立即将当前选中模型加入。
    - 选择已有收藏夹并批量加入。
    - 同一个模型存在于多个收藏夹。
  - 名称校验建议：
    - 必填，去除首尾空白后不能为空。
    - 默认不允许重名，减少筛选和展示歧义。
    - 简介可为空。
  - 首页筛选区后续可增加“收藏夹”分组，用于按收藏夹查看模型；本期至少要保证数据结构与 UI 入口支持后续扩展。
- 批量删除方案：
  - 新增后端接口，例如 `POST /api/models/batch-delete`，请求体为 `{ model_dirs: string[] }`。
  - 后端逐项调用现有 `resolve_model_dir()` 与删除逻辑，汇总成功、失败和错误信息后统一返回。
  - 删除完成后同步从 `favorites`、`printed`、`folders[].modelDirs` 中移除对应目录，避免产生脏引用。
  - 前端确认文案建议显示选中数量，并在数量较多时提示“该操作会物理删除模型目录，无法恢复”。
- 简洁模式方案：
  - 卡片改为更强调封面，封面区域略高于现状，正文区只保留模型名称。
  - 隐藏作者、统计、日期、底部操作按钮等非必要信息。
  - 单卡片操作在简洁模式下建议弱化，但不删除；可通过 hover 或进入多选模式后显示复选能力。
  - 简洁模式仅影响首页卡片展示，不改变接口返回结构。
- 预计改动模块：
  - `app/templates/gallery.html`：增加多选入口、批量操作条、收藏夹弹窗/面板、简洁模式开关。
  - `app/static/js/gallery.js`：增加选中状态管理、收藏夹 CRUD、批量删除调用、简洁模式切换、卡片渲染分支。
  - `app/static/css/gallery.css`：增加多选态、简洁模式、批量操作条、收藏夹弹窗样式。
  - `app/server.py`：扩展 `gallery_flags` 读写结构，新增收藏夹相关接口和批量删除接口。
- 主要风险点：
  - 当前 `gallery.js` 直接用 DOM 拼装卡片，逻辑已经较长；如果继续叠加需求，容易出现渲染与状态同步分叉，需要先整理卡片渲染入口。
  - `save_gallery_flags()` 目前只支持 `favorites` 和 `printed`，直接扩展时要避免覆盖旧数据或写出非法结构。
  - 批量删除若使用前端串行请求，失败处理会很难看；建议一次请求由后端统一处理。
  - 多选模式与现有“点击封面/标题打开详情”存在交互冲突，需要明确优先级。
  - 简洁模式下隐藏信息后，仍要保留必要的可访问性，例如卡片标题、按钮语义、选中状态提示。
- 验证方案：
  - 开启多选后，筛选、搜索、排序、滚动加载都不应清空已选集合。
  - 新建收藏夹后刷新页面，收藏夹仍存在，模型归属正确。
  - 将同一批模型加入已有收藏夹时不应重复写入。
  - 批量删除 1 个、多个、包含不存在目录三种场景都要验证返回与前端提示。
  - 删除后刷新首页，模型消失，且 `favorites`、`printed`、`folders[].modelDirs` 中不存在残留。
  - 切换简洁模式后，卡片只显示封面与模型名称；刷新页面后仍保留当前视图模式。
  - 移动端下多选入口、批量操作条、收藏夹弹窗不应溢出或遮挡主要内容。

  - 新增批量删除接口 `/api/models/batch-delete`，删除后同步清理收藏、已打印和收藏夹引用。
  - 简洁模式改为“图片上悬浮半透明标题”，并在 hover 时滚动显示完整模型名称。
  - 收藏夹入口已调整到左侧分类标签上方，并在选中当前收藏夹时支持“移出当前收藏夹”。
  - 多选按钮在有选中项时增加高亮反馈，降低批量操作状态识别成本。
- 与预期差异：
  - 简洁模式当前按浏览器本地 `localStorage` 持久化，未做服务端全局共享配置。
  - 收藏夹当前支持“创建并加入”“加入已有收藏夹”“按收藏夹筛选”“从当前收藏夹移除模型”，暂未提供重命名和删除收藏夹入口。
