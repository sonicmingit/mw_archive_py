## 1. 用户需求记录（原始需求 + 初步思路）
- 需求描述：
  - 当前项目的图库列表加载速度越来越慢。
  - 现状是每次打开列表都会遍历本地所有模型目录，并逐个读取模型元数据文件。
  - 用户描述中写的是 `mate.json`，但当前代码实际读取的是每个模型目录下的 `meta.json`。
  - 随着模型数量增长，这种“全量遍历 + 全量读取”的方式会持续恶化。
- 用户初步实现思路：
  - 引入尽可能多的方式加快加载速度。
  - 在每次新增归档或导入时同步更新索引，而不是等到下次列表打开时再全量扫描。
  - 增加一个“主动刷新全部索引”的入口，方便在异常或历史数据迁移后手动重建。
- 目标效果：
  - 首页列表加载由“扫描磁盘全部模型目录”改为“直接读取索引”。
  - 正常归档、手动导入、本地批量导入后，图库列表无需全量重扫即可看到新数据。
  - 用户可以手动触发一次全量刷新，用于修复索引、补录旧数据、排查问题。
- 约束/注意事项：
  - 需求当前阶段先做分析记录，不直接开始编码。
  - 需要兼容现有目录结构：`MW_*`、`Others_*`、`LocalModel_*`。
  - 不能影响现有 `meta.json`、`index.html`、附件目录和图片目录的兼容性。

## 2. AI 分析（难度 / 思路 / 改动范围）
- 实现难度：中
- 当前代码现状：
  - 图库接口 `GET /api/gallery` 直接调用 `scan_gallery(CFG)`，位于 [app/server.py](/D:/Code/sonic/1.tampermonkey/bambu/makerworld_archive/0.mw_archive/app/server.py#L2356)。
  - `scan_gallery()` 会遍历 `download_dir` 下全部模型目录，并逐个读取 `meta.json` 后组装列表返回。
  - 当前图库页前端 `load()` 直接请求 `/api/gallery`，位于 [gallery.js](/D:/Code/sonic/1.tampermonkey/bambu/makerworld_archive/0.mw_archive/app/static/js/gallery.js#L440)。
  - 手动导入、归档、本地批量导入都已经是明确的写入链路，分别集中在 [app/server.py](/D:/Code/sonic/1.tampermonkey/bambu/makerworld_archive/0.mw_archive/app/server.py)、[app/archiver.py](/D:/Code/sonic/1.tampermonkey/bambu/makerworld_archive/0.mw_archive/app/archiver.py)、[app/batch_import_service.py](/D:/Code/sonic/1.tampermonkey/bambu/makerworld_archive/0.mw_archive/app/batch_import_service.py)。
  - `batch_import_service.py` 已经存在 `load_local_model_index()` 这种“预加载索引再复用”的模式，说明项目内已经接受“用索引替代反复扫描”的实现思路。
- 性能瓶颈判断：
  - 当前瓶颈不在前端渲染，而在后端列表接口每次请求都进行磁盘全量扫描。
  - 当模型数增加时，瓶颈来自：
  - 根目录遍历成本。
  - 每个模型目录一次 `meta.json` 文件读取和 JSON 解析成本。
  - 每次请求都重复计算封面、摘要、实例数量、发布时间、采集时间等派生字段。
  - 如果模型目录位于机械盘、网络盘、Docker 挂载卷，体感会进一步恶化。
- 推荐总体方案：
  - 为图库引入一份专用索引文件，例如 `download_dir/_gallery_index.json`。
  - 首页列表接口优先直接读取索引文件，不再默认全量扫描全部模型目录。
  - 将“写索引”拆成两类：
  - 增量更新：新增归档、手动导入、本地批量导入、删除模型、编辑模型元数据后，仅更新受影响模型。
  - 全量重建：提供显式接口和页面按钮，必要时重新扫描全目录生成索引。
- 推荐索引内容：
  - 建议索引不是只保存目录名，而是直接保存首页卡片渲染所需字段，避免列表接口二次计算。
  - 每条索引记录建议包含：
  - `dir`
  - `id`
  - `baseName`
  - `title`
  - `source`
  - `cover`
  - `tags`
  - `summary`
  - `author`
  - `stats`
  - `instanceCount`
  - `publishedAt`
  - `collectedAt`
  - `meta_mtime`
  - `indexedAt`
  - 索引文件建议增加 `_meta`：
  - `version`
  - `generatedAt`
  - `itemCount`
  - `fullRebuildAt`
  - 可选的 `schemaVersion`
- 推荐读写策略：
  - 读路径：
  - `GET /api/gallery` 默认读取 `_gallery_index.json`
  - 若索引不存在，则首次自动构建一次
  - 若索引损坏，则返回错误或自动触发一次重建并记录日志
  - 写路径：
  - 新增统一索引模块，例如 `gallery_index.py`
  - 提供 `build_gallery_index(root)`、`load_gallery_index(root)`、`upsert_gallery_index_entry(root, model_dir)`、`remove_gallery_index_entries(root, model_dirs)` 等方法
  - 索引文件写入要采用“先写临时文件再原子替换”的方式，避免中途中断损坏
- 推荐增量更新接入点：
  - 在线归档成功后：
  - 当前 `archive_model_with_lock()` 在最终目录落盘后即可更新该模型索引。
  - 手动导入成功后：
  - 当前 `api_manual_import()` 写完 `meta.json` 和 `index.html` 后即可更新该模型索引。
  - 本地批量导入成功后：
  - 当前 `run_batch_import()` 已经知道是新增模型还是追加实例，导入结束时可批量更新受影响目录索引。
  - 模型编辑后：
  - 当前在线编辑模型元数据、实例导入、附件改动等接口只要改动了 `meta.json` 或会影响列表字段，也应触发索引更新。
  - 模型删除后：
  - 当前批量删除逻辑完成后应同步从索引移除对应项。
- “主动刷新全部”入口建议：
  - 后端新增接口，例如：
  - `POST /api/gallery/rebuild-index`
  - 支持参数：
  - `force`
  - `dry_run`
  - `repair_missing`
  - `cleanup_deleted`
  - 前端在图库页或配置页增加按钮：
  - “刷新列表索引”
  - 按钮点击后显示处理结果，例如总扫描数、索引新增数、更新数、移除数、失败数、耗时。
  - 这个入口是必要的，因为历史数据、手工改目录、外部复制模型等场景不会走正常增量写入链路。
- 可以引入的加速手段：
  - 第一优先级：列表接口改读索引文件，而不是每次全量扫描。
  - 第二优先级：索引项直接存首页所需字段，避免每次请求重新派生。
  - 第三优先级：对单模型索引更新只读取一个 `meta.json`，不要重扫全目录。
  - 第四优先级：全量重建时可并发读取 `meta.json`，但首页正常读请求不依赖并发扫描。
  - 第五优先级：索引项记录 `meta_mtime`，全量重建时可跳过未变更目录。
  - 第六优先级：若后续列表字段增加很多，可考虑把摘要截断、封面路径等派生值提前固化到索引。
- 不推荐的方向：
  - 不建议仅依赖前端缓存，因为新增模型后仍要有稳定的服务端真源。
  - 不建议继续在 `/api/gallery` 中做“边扫边返回”的折中方案，这只是延后问题，不能解决总成本。
  - 当前数据规模下不建议直接引入 SQLite 或外部数据库，索引 JSON 更贴近现有项目结构和维护成本。
- 推荐实现顺序：
  - 第 1 步：补一个独立图库索引模块，能全量构建索引并读取索引。
  - 第 2 步：将 `/api/gallery` 切换为默认读索引。
  - 第 3 步：在归档、手动导入、本地批量导入、编辑、删除链路补增量更新。
  - 第 4 步：增加“重建全部索引”接口和前端入口。
  - 第 5 步：补日志、错误提示和回退策略。
- 预计改动模块：
  - `app/server.py`
  - 新增 `app/gallery_index.py`
  - `app/batch_import_service.py`
  - `app/static/js/gallery.js`
  - `app/templates/gallery.html` 或 `app/templates/config.html`
  - 可能涉及 `app/archiver.py`
- 主要风险点：
  - 增量更新接入点不全，会导致索引与实际目录数据不一致。
  - 用户手工改动模型目录、直接复制目录、手改 `meta.json` 时，索引无法自动感知，需要依赖“主动刷新全部”修复。
  - 如果列表字段依赖非 `meta.json` 文件，而索引只监听 `meta.json` 变化，可能出现局部信息不同步。
  - 索引文件损坏后，`/api/gallery` 需要有明确兜底策略，否则首页会直接不可用。
  - 多线程或并发请求下同时写索引，需要加锁，避免覆盖更新。
- 验证方案：
  - 准备一组包含 `MW_*`、`Others_*`、`LocalModel_*` 的样本数据。
  - 对比“旧版直接扫描结果”和“新版索引读取结果”字段是否一致。
  - 验证新归档一个模型后，无需全量重建即可在列表中看到新卡片。
  - 验证手动导入、本地批量导入、删除模型后索引同步更新。
  - 验证手工删除或复制模型目录后，点击“刷新列表索引”可恢复一致性。
  - 验证索引文件缺失、损坏、部分字段缺失时的兜底行为。
  - 记录首页列表加载耗时，对比改造前后的接口响应时间。
