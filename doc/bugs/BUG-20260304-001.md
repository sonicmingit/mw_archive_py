# [BUG-20260304-001] 特殊字符文件名导致历史归档修复失败

- 状态：✅ 已关闭
- 严重级别：中
- 发现时间：2026-03-04
- 解决时间：2026-03-06
- 影响版本：v5.2.2 及更早历史归档数据
- 解决版本：v5.2.3
- 记录人：sonic
- 最后更新：2026-03-06

## 问题描述
- 部分历史归档模型在执行“一键更新历史归档”后，仍然存在实例文件无法正确匹配的问题。
- 这类模型通常包含特殊字符、路径分隔符、空格变体、中英文混排或历史字段结构差异。
- 典型现象是：
  - `meta.json` 中实例 `fileName` 为空或未被正确修复。
  - 页面下载按钮或打印按钮无法定位到正确的 `instances/*.3mf` 文件。
  - `rebuild_pages.log` 中会出现未定位实例记录。
- 该问题与 GitHub Issue #13 对应，Issue 结论集中在“路径或文件名中有特殊符号导致历史归档修复失败”。

## 影响范围
- 主要影响旧版本归档的数据，尤其是早期使用展示标题、非真实文件名或字段结构不稳定的模型。
- 新归档模型在当前版本下通常不会再复现，因为文件保存逻辑已经改为“展示名称”和“真实文件名”分离。

## 关联现象
- 文件名中包含 `%`、`/`、`\`、全角符号、异常空格等内容时，更容易出现修复失败。
- 示例模型：
  - [MakerWorld 示例模型](https://makerworld.com.cn/zh/models/2187445-zha-bi-xiao-xin-bu-li-bu-li-zuo-wei-men-fei-du-du)
- 历史问题中，页面展示名称可能包含类似：
  - `单色, 0.12mm 层高, 2 层墙, 15% 填充`
- 但实际保存到 `instances` 目录中的真实文件名已经被清洗或按真实文件名字段保存，两者不一致时，历史修复功能无法完全靠推断补齐。

## 复现步骤
1. 使用旧版本归档一个包含特殊字符或标题与真实文件名不一致的模型。
2. 升级到新版本后执行“一键更新历史归档”。
3. 查看对应模型目录下的 `meta.json` 和 `instances` 目录。
4. 可见部分实例的 `fileName` 未能被自动回填，或页面仍无法正确下载对应实例文件。

## 期望结果
- 历史归档修复功能能够自动定位真实实例文件，并为实例补全正确的 `fileName`。
- 页面下载、在线打开和本地实例访问都应使用真实文件名，而不是仅依赖展示标题。

## 实际结果
- 对部分旧数据，历史修复功能无法准确推断出真实文件名。
- 原因不是页面展示名称本身有问题，而是历史落盘文件名与旧版元数据字段之间已经失配。
- 重新归档同一个模型后，问题通常不再出现，因为新版本会直接记录真实文件名。

## 根因分析
- v5.2.3 之前的历史数据中，实例展示名称和实际文件名可能存在不一致。
- 旧数据有时使用 `title` 作为下载或修复依据，而真实文件名实际来自 `name` 或其它字段。
- 当标题中包含特殊字符、路径分隔符、百分号或异常空白时，历史修复逻辑很难仅靠现有字段反推出真实落盘文件名。
- 因此该问题本质上属于“历史数据结构与真实文件落盘结果失配”，而不只是单纯的 `%` 编码问题。

## 当前版本结论
- v5.2.3 已重构文件保存逻辑：
  - 展示名称用于页面展示。
  - 文件名称字段用于真实文件保存与后续下载定位。
- 在当前版本下重新归档出现问题的模型，实测通常不会再复现该问题。
- 但历史修复功能无法完全弥补早期错误归档或字段缺失造成的文件名失配，因此不承诺对所有旧数据自动修复成功。

## 处理方案
可以采用以下两种方式处理：

1. 备份后重新归档
- 先备份修复失败的模型目录。
- 删除或移走异常归档目录后，重新归档该模型。
- 该方式优先推荐，成功率最高。

2. 手动补全 `fileName`
- 打开对应模型目录下的 `meta.json`。
- 找到目标实例节点，查看修复后是否生成了 `fileName` 字段。
- 对比 `instances` 目录中的实际 `.3mf` 文件名。
- 若能确认对应关系，可手动将正确文件名补入实例的 `fileName` 字段。
- 如果 `instances` 目录中本身就没有对应文件，建议直接采用第一种方式重新归档。
